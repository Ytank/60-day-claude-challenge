      How this was built
      System prompt design

      One system prompt runs the entire session in two modes, switched by a control message rather than two separate prompts: during the interview it must return exactly one interview question and nothing else, grounded in the pasted job description and role; on receiving an internal "end session" signal, it switches to returning a single strict JSON performance report instead. Keeping both in one prompt means the model always has full context of what it already asked.

      Why the UI looks this way

      A boardroom aesthetic — charcoal, brass, serif type — matches "professional & polished" better than a generic chat bubble UI, and a running transcript reinforces that this is a formal mock session, not casual chat. The final report reuses the same visual language (score badge, category bars) so the debrief feels like a continuation of the same session rather than a different tool.
